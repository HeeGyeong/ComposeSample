# 작업 진행 워크플로우 (Step-by-Step Spec 우선)

> 이 프로젝트에서 (선택적으로) 작업을 진행할 때 따를 수 있는 spec-우선 방법론입니다.
> 코드를 바로 수정하지 않고, **먼저 작업 내용을 MD(spec) 파일로 작성 → 사용자 확인 → 실행** 순서를 지킵니다.
>
> ⚠️ **CLAUDE.md 우선**: 빌드 실행 주체·커밋·푸시 규칙은 **이 프로젝트의 `CLAUDE.md`를 그대로 따른다.** 이 문서는 그 규칙 위에서 *spec을 어떻게 쪼개고 분석할지*만 다룬다. 충돌이 있으면 항상 `CLAUDE.md`가 우선이다.

---

## 0. 저장 위치

- 모든 작업 spec MD 파일은 **`.claude/specs/`** 폴더에 저장합니다.
- 이 폴더는 `.gitignore`(`.claude/*`)에 의해 **git에 커밋되지 않습니다.** (작업 메모가 코드 커밋에 섞이지 않음 — 추적되는 것은 `.claude/skills`, `.claude/agents`, `.claude/docs`뿐)
- **년-월-주차 폴더** 아래에 작업 단위별 하위 폴더로 정리합니다.

```
.claude/specs/
└── <YYYY-MM-wN>/                ← 주차 폴더 (예: 2026-06-w3)
    └── <작업명>/                 ← 작업 단위별 폴더 (예: feature-collection-literals)
        ├── 00-overview.md       ← 작업 개요 + step 목록 + 진행 현황
        ├── 01-<step명>.md       ← Step 1 상세
        ├── 02-<step명>.md       ← Step 2 상세
        └── ...
```

- **주차 폴더명 규칙**: `YYYY-MM-wN` (월은 제로 패딩 — 문자열 정렬 보장)
  - **주차 계산**: `N = ceil(일/7)` — 1~7일=w1, 8~14일=w2, 15~21일=w3, 22~28일=w4, 29~31일=w5
  - 작업이 주를 넘기더라도 **작업 시작일 기준**으로 폴더를 정하고 이동하지 않는다.
- **작업 폴더명 규칙**: `<유형>-<요약>` (예: `feature-collection-literals`, `fix-router-branch`)

## 1. 진행 원칙

1. **Spec 먼저, 코드 나중**
   - 어떤 작업이든 코드를 건드리기 전에 해당 step의 MD 파일을 먼저 작성한다.
2. **Step 단위로 분리**
   - 작업을 의미 있는 step으로 나누고, **각 step마다 별도 MD 파일**을 만든다.
3. **확인 후 실행 (Gate)**
   - 각 step MD 작성 후 사용자에게 보여주고 **승인(확인)을 받은 뒤** 해당 step의 코드 작업을 실행한다.
   - 승인 없이 다음 step의 코드 작업으로 넘어가지 않는다.
4. **진행 현황 갱신**
   - step을 완료하면 `00-overview.md`의 체크리스트 상태를 업데이트한다.
5. **수정/개선 작업은 영향 분석 필수**
   - 작업 유형이 **기능 추가가 아닌 기존 코드의 수정/개선**(`fix-*`, `improve-*`, 리팩터링, 동작 변경 등)인 경우,
     spec에 **크래시 가능성 / 사이드 이펙트 분석** 섹션을 반드시 포함한다. (템플릿 2-2의 해당 섹션 참고)
   - 분석은 추측이 아닌 **실제 코드 확인에 근거**한다 — 변경 대상의 호출부/참조부를 검색(grep)하고, 영향받는 화면·플로우를 코드로 확인한 뒤 작성한다.
   - **이해 범위가 크면 하네스 사용** — 변경이 다중 파일·실행 경로 추적·크로스모듈 영향처럼 단순 grep으로 근거 확보가 어려운 경우, 멀티에이전트 하네스로 분석의 근거를 확보한다. 목적에 따라 골라 쓴다:
     - `explore-codebase`(`/explore-codebase <대상>`) — **탐색형**. 실행 경로·구조·영향 범위를 병렬 탐색해 *어디가 엮이는지* 파악할 때.
     - `impact-analyze`(`/impact-analyze [대상|diff]`) — **검증형**. 영향 차원별(호출부·생명주기·상태복원·동시성·데이터호환·공유컴포넌트·연계플로우) 리스크를 탐지하고 **adversarial 검증**해 **크래시 가능성/사이드 이펙트 분석 섹션 초안**을 바로 생성할 때. 분석 섹션 작성이 목적이면 이쪽이 직결된다.
     - `design-options`(`/design-options <문제>`) — **결정형**. 접근 방향이 미결정일 때(방향 A/B/C 등) 관점이 다른 설계자 3명이 독립 설계하고 judge panel 이 교차 심사해 **비교표·추천안**을 생성. spec 의 방향 결정 자료로 사용. 방향 결정 후 변경면을 `impact-analyze` 로 넘기는 체인이 자연스럽다.
     - 하네스 모두 토큰 비용이 크므로 **조건부로만** 쓴다. 단일/소수 파일·국소 변경이면 하네스 없이 직접 grep/Read 로 충분하다.
     - 하네스를 돌리기 전 **"<하네스명> 하네스로 영향 분석을 수행합니다"** 한 줄을 사용자에게 고지한다.

## 2. 각 파일 템플릿

### 2-1. `00-overview.md` (작업 개요)

```markdown
# [작업명] 개요

- **이슈/배경**: (요구사항 요약)
- **목표**: (이 작업으로 달성하려는 것)
- **영향 범위**: (수정 대상 파일/모듈 대략)

## Step 목록 & 진행 현황
- [ ] 01. <step명> — (한 줄 설명)
- [ ] 02. <step명> — (한 줄 설명)
- [ ] 03. <step명> — (한 줄 설명)

## 진행 로그
| 일시 | Step | 상태 | 비고 |
|------|------|------|------|
|      |      |      |      |
```

### 2-2. `NN-<step명>.md` (각 step 상세)

```markdown
# Step NN. <step명>

## 상태
- [ ] 작성됨 (spec 검토 대기)
- [ ] 승인됨
- [ ] 구현 완료

## 목적
(이 step에서 무엇을, 왜 하는지)

## 변경 대상
- `path/to/file.kt` — (무엇을 변경)

## 작업 내용 (상세)
1. ...
2. ...

## 검증 방법
- (빌드/동작 확인 방법 — 빌드 실행 주체는 CLAUDE.md 규칙을 따른다)

## 크래시 가능성 / 사이드 이펙트 분석 ※ 수정·개선 작업 시 필수 (기능 추가는 생략 가능)

### 크래시 가능성
| 시나리오 | 원인 (예상 예외) | 가능성 | 대응/방어 |
|----------|------------------|--------|-----------|
| (예: 화면 회전/process death 후 복원) | (예: rememberSaveable 누락 → 상태 유실 / 복원 시 NPE) | 상/중/하 | (예: rememberSaveable + Saver 적용) |

- Compose 환경에서 특히 점검할 항목:
  - **재구성/상태**: `remember`/`rememberSaveable` 키 누락, 불안정 파라미터로 인한 과도한 재구성, 상태 호이스팅 위반
  - **사이드 이펙트**: `LaunchedEffect`/`DisposableEffect` 의존성 키·정리(cleanup) 누락, 콜백 시점
  - **생명주기/스코프**: ViewModel 스코프와 Composable 생명주기 불일치, `rememberCoroutineScope` 취소
  - **상태 복원**: rememberSaveable/Saver, process death 후 복원, Navigation 인자 직렬화 호환
  - **동시성**: Flow/coroutine 취소·재구독, 메인 스레드 밖 UI 접근, StateFlow 수집 시점
  - **데이터 호환**: 서버 응답(Retrofit/Ktor)/로컬 캐시(Room)의 기존 데이터가 변경된 파싱·분기 로직을 통과하는지

### 사이드 이펙트
- **호출부/참조부 영향**: 변경한 함수·프로퍼티·리소스를 사용하는 다른 곳 (grep 결과 기재 — "다른 곳 없음"도 확인 결과로 명시)
- **공유 컴포넌트 영향**: 공통 Composable/extension/테마/공유 ViewModel 변경 시 이를 쓰는 다른 화면 목록
- **기존 동작 변화**: 의도한 변경 외에 달라지는 동작 (이벤트 발행 시점, 기본값, 정렬/필터 결과 등)
- **연계 플로우**: 네비게이션 콜백(onBackEvent), ExampleRouter 라우팅, 공유 상태 등 간접 경로로 영향받는 플로우

### 분석 근거
- (호출부 검색 명령/결과, 확인한 파일 경로 등 — 분석이 코드 확인에 기반했음을 남김)

## 리스크 / 주의사항
- (위 분석 외 일반 리스크 — 기존 경고 무시 여부, 빌드 영향 등)
```

> **작성 기준**: "크래시 가능성 / 사이드 이펙트 분석" 섹션은 **수정/개선 step에서는 비워두거나 생략할 수 없다.**
> 분석 결과 해당 없음이면 "해당 없음"이 아니라 **무엇을 확인했고 왜 문제가 없는지**를 적는다. (예: "호출부 grep 결과 단일 호출 — 영향 범위 해당 ExampleUI 내부로 한정")

## 3. 실행 절차 (Claude가 따르는 순서)

1. 작업 요청을 받으면 오늘 날짜로 주차 폴더를 계산하고 `.claude/specs/<YYYY-MM-wN>/<작업명>/` 폴더를 만든다.
2. `00-overview.md` 를 작성하고 **step을 나눠 제시**한다.
3. 각 step의 `NN-<step명>.md` 를 작성한다. (필요 시 한 번에 모든 step, 또는 step 진행마다 작성)
   - **수정/개선 작업이면** spec 작성 전에 변경 대상의 호출부·참조부를 먼저 조사(grep/Read)하고,
     그 결과를 바탕으로 **크래시 가능성 / 사이드 이펙트 분석** 섹션을 채운다.
     - 이때 **이해 범위가 크면**(다중 파일·실행 경로·크로스모듈) grep/Read 대신 하네스를 쓴다 — 구조/경로 파악은 `explore-codebase`, **분석 섹션 초안 생성은 `impact-analyze`**(검증형). 실행 전 한 줄 고지하고, 반환된 결과(맵/실행 경로/검증된 리스크 섹션)를 분석 섹션의 **분석 근거**에 인용한다.
4. 사용자에게 spec을 보여주고 **확인을 요청**한다. → ⛔ 승인 전까지 코드 수정 금지
5. 승인된 step만 코드로 구현한다.
6. 구현 후 해당 step MD의 상태 체크박스와 `00-overview.md` 진행 로그를 갱신한다.
7. 다음 step으로 이동, 4~6 반복.

> 빌드 검증·커밋·푸시는 위 절차와 별개로 **CLAUDE.md의 규칙(빌드 후 검증, 커밋+푸시 함께)** 을 따른다.

## 4. 작업 중 변경점 처리 (Spec 동기화)

작업을 진행하다 보면 처음 작성한 spec과 달라지는 경우가 생긴다. **코드와 spec MD를 항상 일치시키는 것**이 원칙이며, 아래 두 가지 경우로 처리한다.

### 4-1. 기존 step 내 변경점 발생 (내용이 바뀜)
- 해당 step의 `NN-<step명>.md` 를 **변경점에 맞게 갱신**한다. (변경 대상 / 작업 내용 / 리스크 등)
- 수정/개선 작업에서 **변경 대상이나 접근 방식이 바뀌면**, 크래시 가능성 / 사이드 이펙트 분석도 **바뀐 내용 기준으로 재분석·갱신**한다.
- `00-overview.md` 의 **진행 로그**에 변경 사실을 한 줄 남긴다. (무엇이 왜 바뀌었는지)
- 판단 기준:
  - 변경이 **이미 승인된 범위 안**의 사소한 조정(파일 경로 정정, 표현 수정 등)이면 → 갱신 후 그대로 진행.
  - 변경이 **승인 범위를 벗어남**(동작 변경, 새 파일 수정, 접근 방식 변경 등)이면 → 갱신한 spec을 사용자에게 보여주고 **재승인(Gate)** 을 받은 뒤 진행. ⛔ 재승인 전 코드 수정 금지.

### 4-2. step이 늘어나는 경우 (새 작업 추가)
- 새 step의 `NN-<step명>.md` 파일을 **추가로 생성**한다. (번호는 다음 순번)
- `00-overview.md` 의 **Step 목록**에 새 step을 추가하고 진행 로그에도 기록한다.
- 새 step 역시 **승인 Gate를 적용** — 작성 → 사용자 확인 → 구현.

> 요약 흐름: **변경 감지 → spec MD 갱신/추가 → overview 동기화 → (승인 범위 벗어나면) 재승인 → 구현**

## 5. 프로젝트 규칙 연계

- 빌드·커밋·푸시 규칙: **`CLAUDE.md`** (이 문서보다 우선)
- 코딩 컨벤션 / 예제 추가 절차: `CLAUDE.md`
- 아키텍처/레이어 규칙: 루트 `ARCHITECTURE.md`, `CLAUDE.md` 의 Layer Rules
- 상세 규칙 문서 인덱스: `app/src/main/java/com/example/composesample/docs/README.md`
- 코드베이스 탐색 하네스: `explore-codebase` 스킬(`/explore-codebase <대상>`) — read-only 멀티에이전트로 실행 경로·아키텍처를 병렬 탐색(**탐색형**). (조건부 사용 — §1.5)
- 영향 분석 하네스: `impact-analyze` 스킬(`/impact-analyze [대상|diff]`) — read-only 멀티에이전트로 영향 차원별 리스크를 탐지·**adversarial 검증**해 크래시/사이드이펙트 분석 섹션 초안 생성(**검증형**). (조건부 사용 — §1.5)
- 설계 비교 하네스: `design-options` 스킬(`/design-options <문제>`) — read-only 멀티에이전트로 독립 설계 N개를 judge panel 심사해 비교표·추천안 생성(**결정형**). 방향 미결정 작업의 spec 작성 전 사용. (조건부 사용 — §1.5)
- 유사 버그 전파 탐지 하네스: `find-similar-bugs` 스킬(`/find-similar-bugs <설명|커밋|diff>`) — 수정된 버그의 패턴을 추상화해 코드베이스 다른 곳의 같은 버그를 multi-modal 탐색·adversarial 검증(**전파탐지형**). fix 작업 완료 직후 "다른 데도 있나" 확인용. 탐지까지만 — 발견된 형제 버그의 수정은 별도 작업 spec 으로 진행.
